Key Points Breakdown

Distributed and Decentralized

  • Distributed:
    • Runs on multiple machines, appearing as a unified whole.
    • Optimized for performance across multiple data centers.
  • Decentralized:
    • Every node is identical; no single point of failure.
    • Peer-to-peer architecture using a gossip protocol.
    • Simpler to operate and maintain than primary/secondary stores.

Elastic Scalability

  • Horizontal Scaling:
    • Add more machines to share the load.
    • Seamless scaling up and down without major reconfiguration.
  • Elastic Scalability:
    • New nodes can start participating with minimal disruption.
    • Supports seasonal workloads and platform migrations.

High Availability and Fault Tolerance

  • High Availability:
    • Replace failed nodes without downtime.
    • Replicate data to multiple data centers.
  • Fault Tolerance:
    • Handles hardware failures, network disruptions, and data corruption.

Tuneable Consistency

  • Consistency Definition: A read always returns the most recently written value.
  • Consistency Models:
    • Strict Consistency: Every read returns the latest write.
    • Causal Consistency: Writes that are related must be read in sequence.
    • Weak (Eventual) Consistency: All replicas will eventually be consistent.
  • Tuneable Consistency in Cassandra:
    • Clients can control the consistency level.
    • Balance between consistency and availability.
    • Always writable, conflict resolution during reads.

Brewerâs CAP Theorem

  • Three Requirements:
    • Consistency: All clients read the same value.
    • Availability: All clients can read/write data.
    • Partition Tolerance: System continues functioning despite network segmentation.

CAP theorem indicates that you can realize only two of these properties at once

  • Trade-offs:
    • Only two of the three can be strongly supported.
    • CA: Consistency and Availability.
    • CP: Consistency and Partition Tolerance.
    • AP: Availability and Partition Tolerance.
  • Cassandraâs Position: Focus on availability and partition tolerance.

Row-Oriented

  • Data Model:
    • Partitioned row store with sparse multidimensional hash tables.
    • Efficient storage and query processing with flexible schema support.
  • Schema Evolution:
    • Transition from schema-free to schema-optional to flexible schema with CQL.

High Performance

  • Design: Optimized for multiprocessor/multicore machines.
  • Scalability: Consistent performance across many machines and data centers.

History of Cassandra

  • Origins: Created at Facebook in 2007 for inbox search.
  • Development:
    • Led by Jeff Hammerbacher, Avinash Lakshman, Karthik Ranganathan, and Prashant Malik.
    • Open-sourced in 2008, moved to Apache Incubator in 2009, top-level project in 2010.
  • Community: Contributions from companies like Twitter, LinkedIn, Apple, and independent developers.
  • Evolution:
    • Introduction of CQL for improved usability.
    • Major contributions from enterprises and consultancies.

Is Cassandra a Good Fit for My Project?

  • Large Deployments: Effective for projects needing several nodes or more.
  • High Write Throughput: Optimized for write-heavy applications.
  • Geographical Distribution: Supports data replication across multiple data centers.
  • Hybrid Cloud and Multicloud: Flexible deployment across different providers.
