Monitoring Cassandra with JMX



Java Management Extensions (JMX)

  • Enables remote management of Cassandra nodes.
  • Provides insight into application health and performance.
  • Instrumentation allows JVM to gather data for external tools.

Connecting Remotely via JMX

  • By default, JMX is enabled for local access only.
  • To enable remote access, edit <cassandra-home>/cassandra-env.sh.

JMX Clients

  • Nodetool: Command-line tool bundled with Cassandra.
  • JConsole: GUI client for working with MBeans, included with JDK.
  • Oracle Java Mission Control and Visual VM: Robust metrics and diagnostics tools.
  • MX4J: Open-source JMX implementation with HTTP/HTML interface.
  • Jmxterm: Command-line JMX client.
  • IDE Integrations: JMX clients for IDEs like Eclipse.

Cassandra's MBeans

Managed Beans (MBeans)

  • Special type of Java bean representing a manageable resource.
  • Examples include CompactionManagerMBean, StorageServiceMBean, HintsServiceMBean, etc.

Key MBeans

  • Storage Service MBean: Essential for monitoring and managing the storage engine.
  • Storage Proxy MBean: Handles client requests and inter-node communications.
  • Hints Service MBean: Manages hinted handoff operations.
  • Column Family Store MBean: Provides access to compaction and compression settings.
  • Commit Log MBean: Exposes commit log state and recovery operations.
  • Compaction Manager MBean: Statistics and operations related to compactions.
  • Cache Service MBean: Access to key, row, chunk, and counter caches.

Cluster-Related MBeans

  • Gossiper MBean: Manages node gossip.
  • Failure Detector MBean: Describes states and Phi scores of nodes.
  • Snitch MBeans: Monitor and configure snitch behavior.
  • Stream Manager MBean: Manages SSTable streaming between nodes.
  • Messaging Service MBean: Manages messages between nodes.

Internal MBeans

  • Thread Pool MBeans: Monitor and configure thread pools.
  • Garbage Collection MBeans: Monitor JVM garbage collection.
  • Security MBeans: Configure client authentication caching.
  • Metrics MBeans: Report various application performance metrics.

Monitoring with nodetool

Using nodetool

  • Command-line tool for monitoring and managing Cassandra.
  • Uses JMX to access MBeans.
  • Commands include: describecluster, status, info, ring, tpstats, tablestats.

Common Commands

  • describecluster: Prints basic cluster information.
  • status: Identifies nodes and their states.
  • info: Gets basic data about a specific node.
  • ring: Determines node states in the ring.

Virtual Tables

Introduction

  • Not actual tables stored using Cassandraâs typical write path.

System Virtual Schema

  • Tables: keyspaces, columns, tables.
  • Provides metadata similar to DESCRIBE operations in CQL.

System Views

  • Tables include: caches, clients, disk_usage, settings, sstable_tasks, etc.
  • Useful for identifying issues like wide partitions and tombstone counts.

Metrics

Metrics Overview

  • Metrics are vital for system observability.
  • Cassandra reports metrics at OS, JVM, and application levels.
  • Metrics include: buffer pool, CQL, cache, client, commit log, compaction, connection, dropped message, read repair, storage, streaming, thread pool, table, and keyspace metrics.

Metrics Aggregation

  • Use tools like Prometheus and Grafana for metrics aggregation and visualization.
  • GitHub repositories provide configurations for integrating Cassandra with these tools.

Logging

Logging Configuration

  • Cassandra uses SLF4J API with Logback implementation.
  • Logs include system.log, debug.log, gc.log.

Log Levels

  • Configurable levels: ALL < DEBUG < INFO < WARN < ERROR < FATAL < OFF.
  • Default log files stored in logs directory under Cassandra installation.

Full Query Logging

  • Enabled in Cassandra 4.0.
  • Logs CQL queries for detailed analysis and replay.

Log Aggregation and Distributed Tracing

  • Aggregate logs using tools like ELK stack (Elasticsearch, Logstash, Kibana).
  • Perform distributed traces to analyze system-wide requests.
