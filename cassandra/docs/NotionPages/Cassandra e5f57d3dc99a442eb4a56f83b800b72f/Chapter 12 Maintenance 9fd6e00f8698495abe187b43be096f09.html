Health Check

Key Health Check Procedures

  1. Node Status:
    • Use nodetool status to ensure all nodes are up and reporting normal status.
    • Verify the load column to ensure the cluster is well-balanced.
  1. Dropped Messages and Blocked Writers:
    • Use nodetool tpstats to check for dropped messages and blocked flush writers.
    • Address issues by reducing load, scaling up hardware, or adding nodes.
  1. Logs:
    • Check logs for ERROR or WARN level messages.
    • Address issues such as OutOfMemoryError or configuration errors.
  1. Configuration Review:
    • Verify settings in cassandra.yaml and cassandra-env.sh files.
    • Ensure keyspace and table configurations match the cluster topology.
  1. Network and NTP:
    • Ensure network connectivity and functioning NTP servers.

Additional Health Checks:

  • Experienced Cassandra operators often customize health checks to suit their environment.

Common Maintenance Tasks

Flush

  • Use nodetool flush to force Cassandra to write data from memtables to SSTables:
    shCopy code
$ nodetool flush
  • Optionally, specify keyspaces or tables:
    shCopy code
$ nodetool flush <keyspace>
$ nodetool flush <keyspace> <table>

Drain

  • Similar to flush but stops Cassandra from accepting commands:
    shCopy code
$ nodetool drain

Cleanup

  • Reclaims disk space by removing data no longer owned by a node:
    shCopy code
$ nodetool cleanup

Repair

  • Ensures data consistency across nodes:
    shCopy code
$ nodetool repair
  • Options include limiting to local data center (local), running sequentially (seq), or in parallel (par).

Rebuilding Indexes

  • Rebuild secondary indexes using:
    shCopy code
$ nodetool rebuild_index <keyspace> <index>

Adding Nodes

To an Existing Data Center

  • Install and configure Cassandra on new machines.
  • Ensure the same version and configuration values as existing nodes.
  • Add nodes to each rack evenly.
  • Enable bootstrapping in cassandra.yaml:
    yamlCopy code
auto_bootstrap: true

To a New Data Center

  • Configure appropriate snitch.
  • Select seed nodes and configure accordingly.
  • Update replication options for keyspaces:
    sqlCopy code
ALTER KEYSPACE <keyspace> WITH REPLICATION = {'class': 'NetworkTopologyStrategy', 'DC1': 3, 'DC2': 3};
  • Rebuild data using:
    shCopy code
$ nodetool rebuild -- <source_dc>

Handling Node Failure

Repairing Failed Nodes

  • Determine downtime duration.
  • Use hinted handoff or repair as appropriate.

Replacing Nodes

  • Replace a node using Dcassandra.replace_address_first_boot=<address> JVM option.
  • Monitor with nodetool netstats.

Removing Nodes

  • Decommission an up node:
    shCopy code
$ nodetool decommission
  • Remove a down node:
    shCopy code
$ nodetool removenode <host_id>
  • Assassinate as a last resort:
    shCopy code
$ nodetool assassinate <ip_address>

Upgrading Cassandra

Rolling Upgrade Procedure

  1. Drain the node:
    shCopy code
$ nodetool drain
  1. Stop the node.
  1. Backup configuration files.
  1. Install the new version.
  1. Update configuration files.
  1. Repeat for each node.

Backup and Recovery

Taking a Snapshot

  • Perform a snapshot:
    shCopy code
$ nodetool snapshot
  • List snapshots:
    shCopy code
$ nodetool listsnapshots

Clearing a Snapshot

  • Clear snapshots:
    shCopy code
$ nodetool clearsnapshot

Incremental Backup

  • Enable incremental backup:
    shCopy code
$ nodetool enablebackup

Restoring from Snapshot

  • Collect the latest snapshot and incremental backups.
  • Truncate tables before restoring.
  • Use sstableloader for topology changes or use nodetool import for same topology.
