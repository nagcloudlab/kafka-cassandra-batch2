Configuring and Deploying Cassandra

Chapter Overview

  • Build your first cluster.
  • Configure Cassandra nodes: partitioning, snitches, replication.
  • Deploying Cassandra in production and cloud environments.

Cassandra Cluster Manager

Out-of-the-Box Configuration

  • Cassandra starts with no configuration.
  • Download, decompress, execute to start the server with default settings.

Customizability

  • Emphasis on configurability and customization.
  • Use Cassandra Cluster Manager (ccm) to practice building clusters.
  • https://github.com/riptano/ccm
  • Allows running a multinode cluster on a single machine.

Creating Clusters for Testing

  • Use real clusters for unit and integration testing.
  • Tools: Docker, ccm.
  • Quick start: $ pip install ccm or $ brew install ccm.

Basic Commands

  • ccm create -v 4.0.0 -n 3 reservation_cluster --vnodes
  • ccm list
  • ccm status
  • ccm start

Node Status and Configuration

  • Check status: ccm status, ccm node1 status.
  • Examine configuration: diff node1/conf/ node2/conf/.

Adding Nodes to a Cluster

  • ccm add node4 -i 127.0.0.4 -j 7400
  • Start new node: ccm node4 start.

Dynamic Ring Participation

  • Nodes can be brought down and back up without disrupting the cluster.
  • Example commands: ccm stop, ccm status.

Node Configuration

Cassandra.yaml File

  • Primary configuration file.
  • Key values: cluster name, partitioner, snitch, seed nodes.

Changing Cluster Name

  • Update cluster_name property in cassandra.yaml.

Seed Nodes

  • Contact points for other nodes.
  • Example: seeds: "127.0.0.1, 127.0.0.2, 127.0.0.3"

Snitches

  • Gather information about network topology.
  • Configure with endpoint_snitch property.
  • Types: SimpleSnitch, PropertyFileSnitch, GossipingPropertyFileSnitch, RackInferringSnitch, DynamicEndpointSnitch.

Partitioners

  • Specify how partition keys are mapped to token values.
  • Types: Murmur3Partitioner (default), RandomPartitioner, OrderPreservingPartitioner, ByteOrderedPartitioner.

Network Interfaces

Internode Settings

  • listen_address: IP address for incoming connections.
  • broadcast_address: IP address advertised to other nodes.
  • storage_port: Port for internode communications.

Client-to-Node Settings

  • rpc_address: IP address for native transport (CQL).
  • broadcast_rpc_address: IP address advertised to clients.
  • native_transport_port: Port for native transport (default: 9042).

Deprecation of Thrift RPC Properties

  • Thrift interface deprecated in Cassandra 4.0.

Data Storage

File Locations

  • Default: $CASSANDRA_HOME/data or %CASSANDRA_HOME%/data.
  • Commit log location: commitlog_directory.
  • Datafile location: data_file_directories.

Disk Failure Policies

  • Properties: disk_failure_policy, commit_failure_policy.

Startup and JVM Settings

Configuration Files

  • conf/jvm.options: JVM settings.
  • conf/logback.xml: Logging configuration.

Logging and JMX

  • Examined in detail in subsequent chapters.

Creating a Cluster in Docker

Quick Setup

  • Create Docker network: $ docker network create my-network.
  • Create nodes:
    • $ docker run --name node1 --network my-network -p 9042:9042 -d cassandra
    • $ docker run --name node2 -d --network my-network -p 9042:9042 -d -e CASSANDRA_SEEDS=node1 cassandra
    • $ docker run --name node3 -d --network my-network -p 9042:9042 -d -e CASSANDRA_SEEDS=node1,node2 cassandra

Planning a Cluster Deployment

Topology and Replication Strategies

  • Cluster topology: number of data centers, geographic distribution.
  • Replication strategies: SimpleStrategy, NetworkTopologyStrategy.

Sizing Your Cluster

  • Calculate storage size and required number of nodes.
  • Consider replication factor and compaction strategy.

Selecting Instances

  • Development: at least two cores, 8 GB memory.
  • Production: at least eight cores, 32 GB memory.
  • HDDs vs. SSDs: SSDs preferred for performance.

Network Considerations

  • Ensure sufficient network bandwidth (1 Gbps or higher).
  • Avoid load balancers.
  • Synchronize clocks using NTP.

Cloud Deployment

Advantages of Cloud Deployment

  • High availability and disaster recovery.
  • Improved application response time.

Amazon Web Services (AWS)

  • Use EC2 instances, choose between ephemeral storage and EBS.
  • Configure networking with VPC, Direct Connect, or VPN.

Google Cloud Platform (GCP)

  • Use Google Compute Engine (GCE) for regions and zones.
  • Recommended VMs: n1-standard, n1-highmem.
  • Use GoogleCloudSnitch for network configuration.

Microsoft Azure

  • Use Azure Resource Manager for managing deployments.
  • Recommended VMs: D series, H series.
  • Use GossipingPropertyFileSnitch for network configuration.
