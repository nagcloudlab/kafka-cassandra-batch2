Chapter 1. Beyond Relational Databases

Introduction to Cassandra

  • Objective: Help developers and database administrators understand Cassandra.
  • Comparison: Contrast Cassandra with traditional relational database management systems (RDBMS).

What's Wrong with Relational Databases?

  • Advantages of Relational Databases:
    • Ability to express complex relationships.
    • Popularity and widespread use.
  • Challenges:
    • Scalability issues when usage increases.
    • Slow joins in normalized databases.
    • Transaction locking leading to queuing under heavy loads.
    • Solutions often involve vertical and horizontal scaling, configuration updates, query optimization, caching, and denormalization.
    • ACID properties: Atomicity, Consistency, Isolation, Durability.
    • Two-phase commit (2PC) for distributed transactions can cause higher latency and availability issues.

Scalability Problems

  • Vertical Scaling: Adding more hardware (memory, processors, disks) to a single machine.
  • Horizontal Scaling: Adding more machines to form a database cluster.
  • Challenges with Scaling:
    • Data replication and consistency issues.
    • Complexity in configuration and optimization.
    • Need for distributed caches (e.g., Redis, memcached).
    • Denormalization to improve performance, violating Codd's 12 Rules for relational data.

Sharding and Shared-Nothing Architecture

  • Sharding: Dividing data across multiple machines to handle large data sets.
    • Strategies:
      • Feature-based shard (functional segmentation).
      • Key-based sharding.
      • Lookup table (directory-based sharding).
  • Shared-Nothing Architecture:
    • No centralized state; each node operates independently.
    • Popularized by Google’s Bigtable and MapReduce.
    • Cassandra employs shared-nothing architecture with no central controller.

Web Scale and Relational Databases

  • Relational Databases:
    • Good at solving certain data storage problems.
    • Challenges when scaling for web applications.
    • Need for denormalization and avoidance of distributed transactions.
    • Limitations of traditional RDBMSs in handling web-scale data.
  • NoSQL Solutions:
    • Emerging as natural solutions to manage very large databases.
    • Designed to support horizontal scalability and high availability.

The Rise of NoSQL

  • NoSQL: Nonrelational databases supporting "not only SQL" semantics.
    • Categories:
      • Key-value stores (e.g., DynamoDB, Redis).
      • Column stores (e.g., HBase, Kudu).
      • Document stores (e.g., MongoDB, CouchDB).
      • Graph databases (e.g., Neo4j, DataStax Graph).
      • Object databases (e.g., db4o, InterSystems Caché).
      • XML databases (e.g., BaseX, eXist).
      • Multimodel databases (e.g., Azure Cosmos DB, DataStax Enterprise).

New Relational Architectures (NewSQL)

  • NewSQL: Relational databases designed for scalability and transactional guarantees.
    • Examples:
      • Calvin transaction protocol (e.g., FaunaDB).
      • Google’s Spanner (e.g., Cloud Spanner, CockroachDB).

Summary

  • Relational Databases: Effective but challenged by modern scalability and availability requirements.
  • Cassandra: A viable non-relational database solution for web-scale applications.
  • Decision Making: Evaluate based on specific data needs and application requirements.
