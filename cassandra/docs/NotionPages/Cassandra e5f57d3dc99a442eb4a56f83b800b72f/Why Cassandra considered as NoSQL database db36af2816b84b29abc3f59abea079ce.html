Why Cassandra considered as NoSQL database?

1. Schema Flexibility

  • Dynamic Schema: In traditional relational databases (RDBMS), a fixed schema must be defined upfront, meaning you need to specify the structure of your tables (columns and data types) before you can insert data. Cassandra, on the other hand, allows for a flexible schema. You can add or remove columns on the fly without affecting existing rows. This flexibility is crucial for applications that need to evolve quickly without the overhead of database migrations.

2. Distributed Architecture

  • Peer-to-Peer Model: Unlike RDBMS which often use a master-slave architecture (where a master node handles all writes and propagates changes to slave nodes), Cassandra uses a peer-to-peer model. In this architecture, all nodes in a Cassandra cluster are equal, and any node can handle read and write requests. This design eliminates single points of failure and makes the system more resilient and fault-tolerant.
  • Scalability: Cassandra is designed to scale horizontally. This means you can add more servers (nodes) to your cluster to increase capacity and throughput without downtime or complex reconfiguration. Each node in the cluster shares the workload, making it easy to scale out.

3. Data Model

  • Column-Family Data Model: Cassandra’s data model is based on the concept of a column family, similar to a table in RDBMS, but with significant differences. Each column family contains rows, and each row is identified by a unique key. Rows in the same column family can have different columns, allowing for a more flexible and sparse data structure. This model is inspired by Google Bigtable and differs from the rigid tabular format of RDBMS.

4. Consistency and Availability

  • CAP Theorem: The CAP theorem states that in a distributed data store, you can only achieve two out of the following three guarantees: Consistency, Availability, and Partition tolerance. Cassandra is designed with a focus on Availability and Partition tolerance. It allows for tunable consistency, meaning you can choose your desired level of consistency on a per-operation basis, from eventual consistency (faster, but less consistent) to strong consistency (slower, but more reliable).
  • Tunable Consistency: Cassandra provides various consistency levels for read and write operations, such as ANY, ONE, QUORUM, and ALL. This flexibility allows developers to balance between consistency and performance based on application requirements.

5. Query Language

  • CQL (Cassandra Query Language): CQL is designed to look similar to SQL, making it easier for those familiar with relational databases to learn. However, CQL lacks some SQL features, such as JOINs and complex transactions, which are not practical in a distributed environment like Cassandra’s. Instead, CQL focuses on the operations that are efficient in a distributed setting.

6. Replication and Fault Tolerance

  • Replication Strategy: Cassandra replicates data across multiple nodes to ensure high availability and fault tolerance. The replication factor can be configured per keyspace (a keyspace in Cassandra is similar to a database in RDBMS). This means you can control how many copies of your data are stored in the cluster.
  • Consistency Levels: You can configure the consistency level for each operation, specifying how many replicas must acknowledge a read or write operation before it is considered successful. This allows for a balance between data consistency and system performance.

7. Write and Read Efficiency

  • Write Optimization: Cassandra is highly optimized for write-heavy workloads. Writes in Cassandra are append-only and written first to a commit log for durability, then to an in-memory structure (memtable), and eventually flushed to disk as SSTables (Sorted String Tables). This process ensures high write throughput and durability without the need for disk seek operations during writes.
  • Read Efficiency: While reads in Cassandra can be slower than writes due to potential disk I/O operations, several mechanisms like bloom filters, key caches, and row caches are used to optimize read performance. Bloom filters quickly check whether a partition key is present in an SSTable, reducing the need to access the disk unnecessarily.

8. Scalability and Performance

  • Linear Scalability: Cassandra offers linear scalability, meaning that as you add more nodes to the cluster, the system's capacity and performance increase proportionally. This is critical for applications that require handling increasing loads over time.
  • Decentralized Operation: Since there is no single master node, Cassandra avoids bottlenecks and single points of failure. The decentralized approach ensures that all nodes can handle read and write requests, leading to better load distribution and fault tolerance.

9. Geographical Distribution

  • Multi-Data Center Replication: Cassandra supports replication across multiple data centers, which enhances disaster recovery and improves data locality. This is particularly beneficial for global applications that need to provide low-latency access to data for users in different geographic locations.
